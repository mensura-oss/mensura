# Mensura Studio

Mensura Studio is the Tauri 2 desktop client for the local Mensura Core service. The current vertical slice supports selecting a workspace, inspecting local Git state, building and browsing a deterministic Vault inventory, indexing the workspace into searchable Vault memory with ranked lexical-vector search and a basic architecture summary, selecting exact files into an immutable context pack, running configured Guard checks, creating a task and context-bound queued run, manually executing through Core's deterministic provider or one optional locally configured OpenAI BYOK adapter, and reviewing a separately persisted bounded change proposal. An approved proposal can additionally be verified in a temporary isolated Git worktree where Guard runs against the materialized suggestion, and an approved, verified proposal can then be explicitly applied to the live working tree. Proposal approval/rejection records review only, verification never touches the live branch, and apply-to-live is a distinct high-trust action that writes verified content to the working tree without ever staging, committing, or pushing.

## Requirements

- Node.js 22 or newer
- pnpm 11 or newer
- Rust 1.77.2 or newer
- Mensura Core running on Python 3.12
- Tauri platform prerequisites for your operating system

Install workspace dependencies once from the repository root:

```sh
pnpm install
```

## Run Studio with Core

Start Core in one terminal:

```sh
cd services/core
.venv/bin/python -m uvicorn mensura_core.main:create_sql_app --factory --reload
```

`create_sql_app` is the durable entry point (SQLite persistence, migrations, and the background job worker); the bare `mensura_core.main:app` target is an ephemeral in-memory variant only.

Start the native Studio window from the repository root in another terminal:

```sh
pnpm studio:dev
```

Studio defaults to `http://127.0.0.1:8000`. The frontend base URL can be changed with `VITE_MENSURA_CORE_URL`, but the Tauri capability currently permits only `http://127.0.0.1:8000/**` and `http://localhost:8000/**`. Expanding that native allowlist is a deliberate security change, not a runtime wildcard.

## Checks and builds

From the repository root:

```sh
pnpm check
pnpm studio:build
```

`pnpm check` runs TypeScript checks, frontend/shared tests, production frontend builds, and `cargo check` for the native shell. `pnpm studio:build` creates the desktop bundle for the current platform.

## Current UI

- Static developer-tool shell with sidebar, top bar, and content grid
- Core liveness polling and manual refresh
- Compact provider settings: deterministic availability plus write-only OpenAI key/model configuration and redacted status
- Workspace list, create form, selectable active workspace, and restored local selection
- Compact read-only repository summary for the active workspace: branch/detached state, clean/dirty badge, staged/unstaged/untracked counts, and up to eight changed-path metadata entries
- **Workspace** panel for the active workspace: a collapsible read-only repository file tree (folded from the Vault inventory, with code/doc/config/binary type badges), a read-only Monaco editor that opens a selected file's bounded text with syntax highlighting (TypeScript/JavaScript/Python/Markdown and more), a Git branch/clean-dirty header (or a "Connect a repository" hint for a non-Git workspace), an "Indexed by Vault" status badge, and a three-column Core-backed task board (Backlog / In progress / Done) showing the workspace's real tasks with per-task latest-run badges and a bounded **Start run** action on eligible cards that launches a queued run through the existing context-pack/run flow. A Vault search hit can "Open in Workspace editor" to load the same file and scroll to/outline the matched line range
- Manual Vault inventory build/refresh with included/excluded/text/binary counts, compact language summary, up to 200 deterministic file metadata entries, and a 16 KiB UTF-8 text preview inspector
- Vault memory panel that indexes the workspace into searchable chunks, shows indexed/not-indexed status with last-indexed time and code/doc/config/chunk counts, a retrieval-mode badge showing whether search runs with semantic embeddings or a lexical/fallback model, runs a ranked search with a `sourceType` filter and a per-query strategy chip, opens each hit in a line-numbered read-only file view that scrolls to and highlights the matched line range, and generates a basic architecture summary (modules, technologies, entry points)
- Context-pack builder with up to 500 selectable inventory entries, exact pre-creation path review, selected/preview-limit counters, immutable creation, process-local pack list, and read-only manifest inspection
- Manual Guard panel with latest-result loading, `Run checks`, pass/fail and blocking state, aggregate counts, compact lint/test cards, and collapsed bounded output
- Task creation for the active workspace plus task lookup by UUID
- Context-pack selection and preflight summary before queued run creation from created or looked-up tasks
- Run lookup/details showing the persisted immutable pack id, ownership/inventory identities, schema, file count, and byte summary
- Explicit deterministic/configured-OpenAI selection before manual execution, with visible running state, terminal status/timestamps, provider kind/adapter/model/`review.v2` identity, bounded structured result, and persisted structured failure
- Successful-run proposal creation/reopen, immutable lineage, collapsed bounded file suggestions and digest metadata, plus explicit approve/reject review actions that never apply repository changes
- Approved-proposal verification in a temporary isolated Git worktree, with sandbox/Guard/safe-diff results and explicit live-repository-untouched status
- Explicit apply-to-live of an approved, verified proposal with a visible precondition checklist, per-file digest checks, live Guard outcome, and undo-metadata summary
- Background jobs panel listing durable queued/running/succeeded/failed jobs with type, target, timestamps, result artifact, and last-error summary, refreshed live over SSE, plus a demonstrable "queue backup job" action
- RFC 9457 Problem Details and connection errors shown without losing server fields

The repository panel is independent from the rest of the shell. A missing path, non-Git root, or unsupported repository state is shown as RFC 9457 Problem Details without disabling workspace/task/run actions. It never renders patches or file contents and exposes no Git mutation controls.

The Guard panel is also independent. Before running it, review the workspace's `.mensura/guard.json`: a manual run executes its validated Ruff/pytest argv in the workspace through Core. The UI stays pending while the synchronous request runs, shows configuration/execution problems locally, and never auto-runs checks. Core keeps only the latest completed result in memory.

The Vault panel is manual and read-only. Build inventory traverses the current workspace root with Core's fixed exclusion rules and stores only the latest snapshot in memory. Selecting a text file loads at most 16 KiB; selecting a binary file shows metadata without making a preview request. Sensitive, generated, oversized, symlinked, missing, and unsafe paths are refused through RFC 9457 errors. This is deterministic metadata retrieval, not semantic search or a complete secret scanner.

The **Vault memory** panel (below the inventory panel) surfaces Core's Vault index/search/summary MVP. `Index workspace` triggers a full-replace index of the active workspace (`POST /vault/index`); while it runs the button shows `Indexing…` and a status row explains Core is reading and chunking files. Once indexed (or on load via `GET /vault/indexes/{id}`) the panel shows an `Indexed` badge, the last-indexed time, and counts for memory items, chunks, code/doc/config files, size, and skipped files, plus detected languages. A **retrieval-mode badge** beside the status reads the index's embedding metadata (`summary.embedding`) and states how the index was built: `Semantic search active — <backend> / <model>` (e.g. `ollama / nomic-embed-text`) when a local neural backend produced the vectors, or `Lexical search active` for the local lexical model; an index built before embedding metadata existed reads as a legacy lexical index that should be re-indexed for semantic search. A workspace that has never been indexed is shown as a neutral `Not indexed` prompt (the `vault-index-not-built` 404 is treated as an empty state, not an error). Search submits a query with an optional `code`/`doc`/`config` filter (`POST /vault/search`, top 20) and lists ranked hits with `#rank`, path, source-type/language, line range, score, and a snippet, above which a **strategy chip** reports how those results were actually ranked (`Ranked by semantic similarity`, `Ranked by lexical similarity`, or an amber `Lexical fallback — re-index for semantic search` when the stored index was built by a different embedding backend than the one configured now); selecting a hit opens the file in a read-only **File view**. The file view fetches the memory item (`GET /vault/memory/{id}` — no new endpoint), reconstructs a line-numbered rendering of the file from its indexed chunks, scrolls to and highlights the hit's line block, marks any non-indexed gaps explicitly (`⋯ lines X–Y not indexed ⋯`), and offers a `← Back to results` affordance. A stale hit whose file is gone after a re-index (`vault-memory-not-found` 404) shows a bounded "result is stale" message rather than a raw error. This is a viewer reconstructed from the durable index (not a live re-read of the working tree) and is deliberately not an editor — no editing, syntax highlighting, or multi-file tabs. `Generate summary` renders a heuristic architecture summary (`POST /vault/summarize`): file counts, technologies, entry points, and top-level modules. An "About Vault memory" disclosure states the honest limits: indexing is manual and full-replace (no incremental/watch), search ranks by **vector similarity — semantic with a local embedding backend, otherwise a local lexical model** (hashed term frequencies, with a small exact-substring boost) as shown by the retrieval-mode badge, re-index after the embedding backend changes to refresh vectors, and large or binary files are skipped by design. All errors render as bounded RFC 9457 details.

The **Workspace** panel is the everyday repository surface for the active workspace and is keyed by the active workspace so its state resets cleanly when you switch. Its repository tree and editor reuse existing Core endpoints; the task board adds one small read-only listing (`GET /api/v1/workspaces/{id}/tasks`), the only new Core surface for the board. `Build file tree` runs the deterministic Vault inventory (`POST /vault/inventory`); the returned flat listing (`GET /vault/files`, up to 500 entries) is folded client-side into a collapsible directory tree with small code/doc/config/binary badges. Selecting a file opens its bounded UTF-8 preview (`GET /vault/files/content`, ≤16 KiB) in a **read-only Monaco editor** with extension-based syntax highlighting; `monaco-editor` is bundled locally (offline, no CDN) and lazy-loaded so the initial Studio bundle is unaffected. A status bar shows the workspace's Git branch and clean/dirty state (or a clear "Connect a repository" hint when it is not a Git repository) and an "Indexed by Vault" badge derived from the Vault index status (distinct from the inventory). From the Vault memory panel, a search hit's read-only file view offers **Open in Workspace editor**, which routes the path and matched line range into this panel, scrolls it into view, reveals the file in the tree, and outlines the matched lines in the editor (with an honest note when those lines fall beyond the loaded preview). Binary, too-large/excluded, and truncated files are handled with explicit messages, and a not-yet-built inventory shows a build prompt rather than an error. This is a **viewer, not an IDE**: no editing, saving, multi-file tabs, SCM actions, or IntelliSense. A three-column **task board** (Backlog / In progress / Done) shows the workspace's **real Core tasks** from `GET /api/v1/workspaces/{id}/tasks` (oldest-first), collapsing the eight `TaskStatus` values into the three columns while each card keeps its exact status badge plus, when the task has run, a compact latest-run badge (`queued`/`running`/`succeeded`/`failed`). The board is read-oriented, with one write affordance: a bounded **Start run** dispatch action on each card. A card is **eligible** to launch a run only when its task status is `draft` or `ready` **and** no run is already in flight (`queued`/`running`) for it; eligible cards show an enabled `Start run`, and every other card shows a disabled `Start run` with a short reason (e.g. "This task is `approved` and cannot start a new run." or "A run is already `queued` for this task."). `Start run` opens a compact immutable-context-pack picker and reuses the **existing** launch flow — the same `listContextPacks` query and `createRun` mutation (`POST /api/v1/tasks/{id}/runs`) the Tasks panel uses — so a launched run is an ordinary queued run bound to the selected immutable pack, not a board-specific execution path. While the request is in flight the button reads "Starting…" and is disabled; on success the card shows "Run queued." and the board is refetched so the task's `latestRun` badge appears (and the in-flight guard then disables further launches); a failure renders bounded RFC 9457 Problem Details on the card (e.g. a missing/mismatched pack) and keeps the picker open to retry or cancel. Provider execution stays a separate manual action in the Run panel. Otherwise the board is still read-only — no task create, edit, status transitions, or drag-and-drop. Status→column mapping: Backlog = `draft`/`ready`, In progress = `running`/`review`, Done = `approved`/`rejected`/`failed`/`cancelled`.

The context-pack panel uses that latest inventory but has its own candidate query so it can expose up to the API's 500-file listing limit without colliding with the Vault inspector cache. Before creation it shows every selected path, text-preview versus binary-metadata policy, file count, and a conservative bounded-preview estimate. Core remains authoritative for validation. After creation Studio immediately opens the exact returned locked manifest and shows pack/inventory ids, counts, paths, capture/truncation metadata, and content digests; it intentionally does not dump captured preview bodies into the main shell.

Every task run action loads immutable packs using the task's own workspace id. The user must explicitly select one; Studio shows its full digest and compact file/byte evidence before enabling `Start run`. Core revalidates ownership and persists the binding. Created and looked-up run details make that exact immutable execution context visible; selection is preserved if creation returns Problem Details.

Queued run details expose a provider selector and separate `Execute run` action. Deterministic is the visible default and remains usable without provider discovery; OpenAI is disabled until Core reports it configured. The selected provider kind, prompt version, and model are visible before execution. While the synchronous request is pending, Studio shows running state and disables the action. Core remains authoritative: success replaces/refetches immediately, and failure also refetches because a terminal failed record may have been persisted before an RFC 9457 response. Runs observed in `running` poll once per second until terminal. Terminal runs never show the action again.

Execution review keeps input and output distinct. The immutable pack binding stays in base run details; execution shows provider/kind/adapter/version/model/prompt identity, duration, bounded task summary and interpreted intent, context aggregates/languages, warnings, recommended next steps, or the persisted safe failure. Studio does not render prompts, context preview bodies, raw provider responses/logs, credentials, patches, or repository contents.

Successful execution also exposes a distinct write-isolated proposal section. Studio first checks the workspace proposal collection so reopening a run restores its artifact; otherwise `Create proposal` materializes the run's already validated draft idempotently. The review shows proposal/run/context/provider/prompt lineage, summary and rationale, and file-level create/modify/delete metadata. Suggested text is collapsed by default, constrained to a scrollable region, and accompanied by before/after digests plus stored/original byte and truncation metadata. Approve and Reject are terminal artifact decisions. Neither action writes files or invokes Git, and the UI states that boundary before and after review.

Once a proposal is approved, a separate verification section appears so approval and verification remain distinct decisions. `Verify in isolated sandbox` asks Core to materialize the proposal inside a temporary detached Git worktree of `HEAD`, run the workspace's configured Guard checks there, and remove the sandbox. The latest artifact shows passed/failed status, a closed outcome (`sandbox_verified`, `guard_failed`, or `materialization_failed`), the verified commit and cleanup state, per-file applied/refused results with digests, safe diff aggregates, and collapsed Guard checks with bounded output excerpts—never raw patch dumps or full logs. The section states explicitly that the live branch and working tree are never written, and repeated verifications keep earlier attempt counts visible. Verification problems (unapproved proposal, non-Git workspace, truncated content, sandbox failure) appear as RFC 9457 details inside the section.

Below verification, a distinct live-application section makes apply-to-live an explicit high-trust action. Its boundary copy states that applying writes verified content to the live working tree while still not committing, staging, or pushing, and with no provider involved. A visible precondition checklist (approved, verified, verification passed, not yet applied) gates the action; `Apply to live working tree` is offered only when a passing verification exists and names the exact verification it will use. The result panel shows the closed status (`applied_guard_passed`, `applied_guard_failed`, `applied_guard_unavailable`, or `application_failed`), the live-target commit and HEAD-moved flag, a per-file digest-checked write list, the live Guard outcome (or the reason Guard could not run), and a compact undo-basis summary—prior digests and restorability, never raw prior file bodies. Application is single-use per proposal, and RFC 9457 problems (live drift, unsafe path, missing/not-passing verification, already applied, or a `workspace-write-in-progress` conflict when another live-tree writer holds the workspace) render inside the section without disabling the action.

The Local BYOK panel sends a key only on explicit save. Core stores it in the OS credential backend and never returns it; Studio does not put it in localStorage or a config file, preserves it in the form only when save fails, and clears the field after success. The non-secret model ID is shown after redacted provider discovery. Saving does not validate the credential with a paid call; a rejected key is surfaced on selected execution. OpenAI network/model use is optional, and a failed real execution never silently switches providers.

The active workspace ID persists in localStorage, and Core data is now durable across restarts via SQLite persistence.

### Live status updates via Server-Sent Events (SSE)

Studio subscribes to Core's SSE endpoint (`GET /api/v1/events/stream`) on startup with an optional workspace filter. When Core publishes lifecycle events (`run.status.changed`, `verification.created`, `application.created`, `undo.created`, `backup.created`, `job.status.changed`), Studio receives them through an `EventSource` connection and invalidates the relevant TanStack Query cache keys, triggering fresh REST fetches without waiting for polling ticks.

The event model is compact — each event carries only `entityId`, `status`, and a bounded `summary` (≤200 chars). No file contents, artifact bodies, diffs, or patches are streamed over SSE. REST remains the authoritative source of truth; SSE is a notification mechanism only. On disconnect, `EventSource` auto-reconnects, and any screen can be manually refreshed to recover state.

Existing polling behavior on runs (1-second interval while `running`) is preserved as a reliable fallback, but SSE events now trigger faster updates when available.

The app does not start Core itself. Execution, proposal review, verification, and apply-to-live are manual, synchronous, and bounded; only apply-to-live writes the workspace, and it never invokes Git. OpenAI is the only optional real adapter and deterministic remains first-class. Verification, apply, undo, and backup can additionally run as durable queued jobs drained by an in-process Core worker. A read-only Monaco file viewer, a read-only repository tree, and a **Core-backed task board** (real workspace tasks with latest-run badges) with a bounded from-board **Start run** dispatch action landed in these slices; still intentionally deferred are Git writes (stage/commit/push/checkout/reset), additional vendors/models UX, arbitrary prompt editing, credential read-back, automatic job retries, job cancellation, external brokers/distributed workers, WebSockets, bidirectional events, guaranteed durable event replay, orchestration, terminals, in-editor file editing and save, multi-tab editors, editor IntelliSense/refactoring, SCM actions from the tree, task create/edit/status transitions and drag-and-drop from the board (only `Start run` launches work; provider execution stays in the Run panel), live-updating (poll/stream) task/run lists, full diff/patch editing, the full Guard policy engine/history/config editor, Hub, plugins, and authentication.
